## 1. 概述

### 1.1 设计目标

- 实现高性能、低延迟的反向隧道代理
- 全程不解析业务协议，保持协议无关性
- 隧道层 TLS 加密，保证传输安全
- 支持多客户端负载均衡，具备横向扩展能力
- 零外部依赖，单二进制可执行

### 1.2 设计原则

- **最小化**：每个组件只做一件事
- **不可变传输**：业务字节从进入到离开全程不修改
- **fail-fast**：无可用节点时立即失败，不排队等待
- **goroutine 无泄漏**：每个连接生命周期结束时保证 goroutine 退出

## 2. 系统分解

### 2.1 模块划分

```
ts/
├── proto/          # 帧协议编解码（共享包）
├── tlsutil/        # TLS 配置与自签证书生成（共享包）
├── server/         # ts-server 主程序
│   ├── tunnel.go   # 隧道监听与客户端管理
│   ├── proxy.go    # 业务端口监听与请求分发
│   ├── lb.go       # 负载均衡（最少连接）
│   └── main.go     # 入口、参数解析、启动
└── client/         # ts-client 主程序
    ├── tunnel.go   # 隧道连接与帧收发
    ├── forward.go  # 本地服务转发
    └── main.go     # 入口、参数解析、启动
```

### 2.2 模块职责

| 模块 | 职责 |
|------|------|
| `proto` | 定义帧结构常量，提供 `WriteFrame` / `ReadFrame` |
| `tlsutil` | 生成自签名证书，构造 `tls.Config` |
| `server/tunnel` | 接受 ts-client 连接，维护客户端池，读帧分发响应 |
| `server/proxy` | 接受 nginx TCP 连接，分配 req_id，驱动双向传输 |
| `server/lb` | 最少连接选择算法，线程安全的节点注册/注销 |
| `client/tunnel` | 连接 ts-server，发送心跳，读帧分发请求 |
| `client/forward` | 对每个 req_id 建本地 TCP 连接，双向透传 |

## 3. 模块接口设计

### 3.1 proto 包

```go
// 帧类型常量
const (
    TypeHeartbeat    = 0x01
    TypeHeartbeatAck = 0x02
    TypeDataReq      = 0x03  // server → client（业务请求）
    TypeDataRsp      = 0x04  // client → server（业务响应）
    TypeClose        = 0x05
)

// Frame 帧结构
type Frame struct {
    ReqID   uint32
    Type    uint8
    Payload []byte
}

// WriteFrame 写帧到 io.Writer（内部加写锁由调用方保证）
func WriteFrame(w io.Writer, f *Frame) error

// ReadFrame 从 io.Reader 读一帧
func ReadFrame(r io.Reader) (*Frame, error)
```

### 3.2 tlsutil 包

```go
// GenerateSelfSigned 生成内存中的自签名 RSA-2048 证书，返回 PEM 编码的 cert/key
func GenerateSelfSigned() (certPEM, keyPEM []byte, err error)

// ServerConfig 构造服务端 TLS 配置
func ServerConfig(certPEM, keyPEM []byte) (*tls.Config, error)

// ClientConfig 构造客户端 TLS 配置
// caPEM 为 nil 时且 skipVerify=true，跳过证书验证
func ClientConfig(caPEM []byte, skipVerify bool) (*tls.Config, error)
```

### 3.3 server/lb 包

```go
// Node 代表一个已连接的 ts-client
type Node struct {
    ID          string        // 唯一标识（RemoteAddr）
    conn        net.Conn      // TLS 连接
    writeMu     sync.Mutex    // 写帧互斥锁
    activeConns int32         // 原子计数：当前活跃连接数
    pending     map[uint32]chan *Frame  // req_id → 响应通道
    pendingMu   sync.RWMutex
}

// Pool 负载均衡节点池
type Pool struct {
    mu    sync.RWMutex
    nodes []*Node
}

func (p *Pool) Add(n *Node)
func (p *Pool) Remove(id string)
func (p *Pool) Pick() (*Node, error)   // 最少连接选择，无节点返回 error
func (p *Pool) Len() int
```

## 4. 核心流程设计

### 4.1 ts-server 启动流程

```
main()
  ├── 解析命令行参数
  ├── 加载/生成 TLS 证书
  ├── 初始化 Pool（负载均衡池）
  ├── go 启动隧道监听协程（:9000）
  └── 启动业务监听（:9001，阻塞）
```

### 4.2 ts-client 接入流程（服务端视角）

```
Accept TLS conn
  ├── 创建 Node（含 pending map）
  ├── Pool.Add(node)
  └── go handleNode(node)
        └── loop: ReadFrame(node.conn)
              ├── TypeHeartbeat   → WriteFrame(TypeHeartbeatAck)
              ├── TypeDataRsp     → node.pending[reqID] <- frame
              ├── TypeClose       → close node.pending[reqID] channel
              └── EOF/error       → Pool.Remove; close all pending channels
```

### 4.3 nginx 请求处理流程（服务端视角）

```
Accept TCP conn (nginx)
  ├── Pool.Pick() → 选最少连接节点 node
  ├── atomic.Add(&node.activeConns, +1)
  ├── reqID = atomic.AddUint32(&globalID, 1)
  ├── node.pending[reqID] = make(chan *Frame, 32)
  ├── go nginx→tunnel: loop 读 nginx conn → WriteFrame(TypeDataReq, reqID, data)
  ├── go tunnel→nginx: loop 读 node.pending[reqID] → 写 nginx conn
  └── 任一方向结束：
        WriteFrame(TypeClose, reqID)
        delete node.pending[reqID]
        atomic.Add(&node.activeConns, -1)
        关闭 nginx conn
```

### 4.4 ts-client 处理流程

```
main()
  ├── 解析参数
  └── loop（断线重连）:
        conn = tls.Dial(serverAddr)
        go heartbeat(conn, interval)
        loop: ReadFrame(conn)
          ├── TypeHeartbeatAck → 忽略（重置超时）
          ├── TypeDataReq      → go handleRequest(conn, reqID, payload, localAddr)
          └── TypeClose        → 关闭对应本地连接
        断线 → 指数退避 sleep → 重连

handleRequest(conn, reqID, data, localAddr):
  ├── net.Dial(localAddr) → localConn
  ├── 写 data 到 localConn
  ├── go local→tunnel: loop 读 localConn → WriteFrame(TypeDataRsp, reqID, data)
  ├── go tunnel→local: 等待 pending channel 数据写入 localConn（由主循环投递）
  └── 任一方向结束 → WriteFrame(TypeClose, reqID) → 关闭 localConn
```

## 6. 并发模型

| 场景 | goroutine 数量 |
|------|---------------|
| 每个 ts-client 隧道连接 | 1（读帧循环）|
| 每个 nginx 业务连接 | 2（nginx→tunnel, tunnel→nginx）|
| 每个 ts-client 本地请求 | 2（local→tunnel, tunnel→local）|
| 心跳定时器 | 1（ts-client 全局）|

**写帧并发安全**：每个 Node 的 `writeMu sync.Mutex` 保证同一隧道连接上多个 goroutine 写帧不会交叉。

## 7. 错误处理策略

| 错误场景 | 处理方式 |
|----------|---------|
| 无可用 ts-client | 立即关闭 nginx 连接，记录 warn 日志 |
| 隧道写帧失败 | 视为连接断开，触发断线处理流程 |
| 本地服务连接失败 | 发 CLOSE 帧，记录 error 日志 |
| 请求超时（60s）| 发 CLOSE 帧，关闭两端连接 |
| ts-client 断线 | 清理所有 pending channel，等待重连 |
| TLS 握手失败 | 拒绝连接，记录 warn 日志 |
